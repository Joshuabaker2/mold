Docker CI / CD flow with Mold

This guide explains where mold sits in the development lifecycle.

Create a branch

When you're working on a project, different features are ready a different times. Branching helps manage this workflow.

Add commits and push

Once your branch has been created, it's time to start making changes. Whenever you add, edit, or delete a file, you're making a commit, and adding them to your branch. This process of adding commits keeps track of your progress as you work on a feature branch.

Once changes have been committed, you push the code back to the repo.

Test, Build, Package, Publish

Once you've committed and/or pushed your code, you run mold which will then test, build, package and publish your app based on the configuration file.

Open a Pull Request

Pull Requests initiate discussions about your commits. This allows for reviewing and resolving any concerns regarding the request.

Discuss and review your code

Once a Pull Request has been opened, the person or team reviewing your changes may have questions or comments.

Deploy

Once your pull request has been reviewed and the branch passes your tests, you can deploy your changes to verify them in production. If your branch causes issues, you can roll it back by deploying the existing master into production.

Merge

Now that your changes have been verified in a non-production environment, it is time to merge your code into the master branch.

Once merged, mold would be run again which whould generate the final images used in production.
